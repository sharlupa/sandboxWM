## Соавторство с ИИ / Разработка с помощью ИИ
Этот проект создается в соавторстве и при активной поддержке нейросетей (AI coding assistants). Все архитектурные решения, исправления ошибок рендеринга и логика работы оконного менеджера разрабатываются человеком в режиме парного программирования с искусственным интеллектом.

## Смысл проекта

Обычные оконные менеджеры — это жёсткие, предсказуемые и скучные. Окна — это просто прямоугольники, которые можно перетаскивать, резайзить и складывать стопкой. sandboxWM ломает этот шаблон.

Идея в том, чтобы превратить рабочий стол в **физическую песочницу**. Окна перестают быть мёртвыми прямоугольниками — они обретают вес, инерцию, гравитацию. Новое окно **падает** сверху и приземляется на пол или на другие окна. Можно **нарисовать** полку курсором — и окна встанут на неё. Можно **склеить** два окна вместе — и они станут одним целым. Можно **разрезать** приложение на части — и работать с кусками по отдельности.

 sandboxWM — это эксперимент на стыке WM и физического движка. Ответ на вопрос: *«что если окна — это не прямоугольники, а физические объекты?»*. Рабочий стол становится не инструментом управления окнами, а **игровой площадкой**, где пользователь сам решает, как выглядит его пространство.

Подробное описание финального видения — в [`sandboxWM_concept.md`](./sandboxWM_concept.md).

> ⚠️ **Статус:** ранняя версия (`0.1.0`). Сейчас реализован базовый тайловый WM на Smithay с оптимизированным рендером и программным курсором. Физический движок, рисование, деформации и слайсинг — это **цель**, а не готовый функционал (см. раздел «Текущая реализация vs. финальная концепция»).

## Финальная концепция (из `sandboxWM_concept.md`)

### 1. Бесконечный тайлинг и гравитация
Рабочий стол — не прямоугольник, а бесконечное пространство с гравитацией и физическим полом. Новое окно падает сверху, пока не ударится об пол или другие окна. У окон есть вес и инерция.

**Режимы гравитации, переключаемые на лету:**
1. Стандартный — гравитация включена.
2. Полная её остановка → окна плавно выстраиваются в строгий тайлинг.
3. Гравитация выключена для **одного** окна — оно висит в воздухе, остальные продолжают падать.
4. Полная «невесомость» — 0 гравитации для всех окон.

*Технически:* каждое окно — `RigidBody` в rapier2d; смена типа тела `Dynamic` → `Kinematic`/`Fixed` фиксирует окно.

### 2. Интерактивное рисование (физические линии)
Линии, нарисованные курсором поверх рабочего стола, становятся физическими препятствиями. Материалы линий:
- **Сталь (пол)** — твёрдая полка, на которую складывают окна.
- **Клей** — липкая линия, окно намертво прилипает; оторвать можно с усилием.
- **Батут** — упругая линия, от которой окна отскакивают.
- **Веревка** — динамическая, сама падает под гравитацией, на неё можно подвесить окно.

*Технически:* цепочка коллайдеров по точкам курсора; `friction` → эффект клея, `restitution` → батут; верёвка — короткие тела, соединённые `BallJoint`.

### 3. Персистентность (сохранение сессии)
При выходе/перезагрузке всё рабочее пространство сохраняется: нарисованные полки/батуты/верёвки и окна с их координатами и физическими свойствами. При новом запуске всё восстанавливается на тех же местах.

*Технически:* `serde` → JSON/TOML файл на выходе; на старте — чтение и восстановление геометрии в rapier2d.

### 4. Скрепление окон (Joints)
- **Сварка** — два окна склеиваются в монолит, двигаются и падают вместе (`WeldJoint`).
- **Пружина** — окна связаны пружиной, оттянутое окно возвращается с ускорением (`SpringJoint`).
- **Веревка** — физическая связь; по умолчанию проходит сквозь скреплённые окна.

### 5. Полная деформация окон (мягкие тела)
Окна могут гнуться в треугольник, круг, смяться в желе при ударе. Текст и UI внутри гнутся вместе с формой. Степень мягкости/желейности настраивается.

*Технически:* окно отрисовывается как 3D-сетка (mesh) через wgpu/glow; физика мягких тел смещает вершины; шейдер натягивает исходную текстуру приложения на деформированную сетку.

### 6. Умное выделение искривлённого текста
Даже на смятом в дугу окне текст выделяется ровно по изогнутой строке.

*Технически:* приложение рисует в скрытом плоском буфере; композитор делает обратное проецирование экранных координат курсора через искажённую UV-сетку в исходные плоские координаты.

### 7. Разрезание окон на части (слайсинг)
Инструмент «нож» режет одно приложение (например, Telegram) на части — список контактов и чат становятся двумя независимыми физическими окнами, приложение при этом работает как единое целое.

*Технически:* протокол `wp_viewporter`; приложение отдаёт один буфер, композитор создаёт два тела и кадрирует текстуру; клики транслируются с поправкой на offset.

## Архитектура (текущая реализация)

### Состояние приложения (`AppState`)
Центральная структура в `state.rs`: Smithay-состояния (Compositor, Shm, XdgShell, Output, Seat), `Space<Window>`, BSP-дерево тайлов, TTY-сессия (`Option<LibSeatSession>`), позиция курсора, флаг `needs_render` для damage-gating (рендер только при реальных изменениях).

### Тайловое дерево (`TileNode`)
```
TileNode::Leaf(Window)
TileNode::Split { dir, ratio, left, right }
```
Направление разбиения выбирается по соотношению сторон ячейки; при удалении листа родительский `Split` схлопывается.

### Управление
| Комбинация | Действие |
|------------|----------|
| `Super + Enter` | Запустить терминал `kitty` |
| `Super + Q` / `Esc` | Завершить работу WM |
| `Super + W` | Закрыть активное окно |
| `Super + ← / → / ↑ / ↓` | Переключить фокус |
| `Super + ЛКМ + drag` | Изменить пропорции тайла |
| ЛКМ по окну | Установить фокус |
| `Ctrl + Alt + F1..F12` | Переключить VT (только TTY/DRM) |

## Сборка и запуск

```bash
cargo build --release
```

Режим определяется автоматически по переменным `WAYLAND_DISPLAY` / `DISPLAY`:

```bash
# Вложенный режим (из графической сессии):
cargo run --release

# Нативный режим (из TTY, требует libseat, libdrm, libgbm, драйвер EGL):
cargo run --release
```
